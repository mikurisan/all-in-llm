传统 RAG 在处理复杂 query 时, 依赖非结构化的 text vector retrieval 的方式, 往往难以捕捉 entity 之间的深层关系. 为了解决这一问题, 将结构化知识图谱融入 RAG 中, 通过利用知识图谱的显式语义关系和图结构优势, GraphRAG 能够提供更精准的 context retrieval 和更强的推理能力, 在多跳查询和事实性要求较高的场景中表现更为出色.

## 1 从传统 RAG 到 Graph RAG 的演进

### 1.1 传统 RAG 框架固有的局限性

基于非结构化 text vector 检索的核心机制存在一些关键局限.

- 关系理解的缺失: 关注于语义相似性, 难以捕获 entity 之间复杂和隐含的关系. 若 query 中涉及多 entity 关联或因果推理时, 检索到的 chunks 之间可能缺乏逻辑联系.

- 上下文的碎片化: 被拆分的 chunks 破坏了原文的结构和 context 连续性. 对于涉及跨多 document 和长距离 text 的场景, 往往力不从心.

- 检索噪声与幻觉风险: 检索可能返回不相关的 document 或仅部分相关的噪声信息, 对 LLM 造成干扰.

- 推理能力有限: 受限于检索到的 documents, 难以支持需要结构化知识进行导航的多跳推理.

- 跨文档联结能力弱: 跨 document / chapter 的 entity 共现与隐式引用仍难以被显示建模, 导致需要连边才能回答的 query 召回不足.

- 实体歧义与别名问题: 仅依赖 vector similarity 难以稳定区分同名 entity.

- 时效性与版本一致性不足: chunks 往往缺少可计算的时间属性与有效期边界, 导致时序敏感 query 难以得到一致 answer.

### 1.2 知识图谱赋能 RAG 的核心优势

- 结构化语义表达: 以图的形式直接编码 entity 之间的显式语义关系, 避免从 text 中进行隐式, 可能存在偏差的推断.

- 增强推理能力: 图的结构天然支持多跳推理, 系统可以沿着路径遍历, 发现间接但关键的知识关联.

- 事实性与可解释性: 可追溯 answer 在图中的推理路径, 为 answer 提供了事实依据.

- 异构数据集成: 可以无缝集成不同来源的结构化或非结构化数据.

更进一步:

- 语义模式和本体支撑: 通过模式 (schema) 和本体 (ontology) 明确 entity 类型, 关系约束与属性域值, 约束推理空间, 降低歧义.

- 溯源与置信度管理: 边与节点可携带来源, 时间戳与置信度, 便于在生成时进行来源归因与冲突消解.

- 时间与版本建模: 支持时间态边与版本化节点, 允许对“历史状态”与“有效区间”进行查询 (Time-travel Query).

### 1.3 GraphRAG: 一种范式革新

GraphRAG 将 retrieval 转变为从知识图谱中寻找相关的实体, 关系, 路径或子图, 将 RAG 技术从“信息检索”向“知识利用”的范式演进.

## 2 GraphRAG 框架的核心架构与工作流程

### 2.1 通用架构 3 阶段

从原始 data 中构建一个高质量的知识图谱:

- 知识抽取: 利用 LLM 或 IE 管线从非/半结构化 text 中抽取实体, 关系与属性, 包含指代消解, 别名归一 (normalization) 与术语标准化.

- 质量控制: 对 3 元组进行置信度评估, 人机协同抽检与冲突消解; 必要时通过规则与本体约束做一次性校验.

- 图谱融合: 进行实体对齐与去重 (entity resolution), 合并跨来源知识并保留来源, 时间戳等溯源信息.

- 存储与索引: 落地到图数据库, 并建立必要的属性与关系索引, 便于后续高效检索与遍历.

根据 query 执行复杂的图谱检索操作, 混合检索策略是主流趋势:

- 实体定位: 先用实体链接或向量检索在图中锁定核心实体节点.

- 子图探索: 从命中节点开始, 利用图查询语言或遍历算法, 进行邻域拓展, 路径发现与约束过滤.

- 结构化证据抽取: 将路径与关键节点属性序列化为可读证据片段, 或生成子图的文本摘要.

- 高级检索: 如 GraphRAG 采用社区检测生成多层次摘要, 实现“全局-局部”联合检索.

增强生成是最后一步, 将检索到的结构化知识与 query 一同注入 prompt. 实践要点:

- 提示设计: 明确要求 model 引用“图证据”, 并在回答中给出来源与路径.

- 证据融合: 将“结构化3元组/路径“与“原片段“联合提供, 兼顾事实性与文本细节.

- 约束回答: 对敏感领域可采用模板化输出与字段校验, 减少 hallucination.

### 2.2 方法论分类

大致可分为 3 类, 反应了知识图谱与 RAG 结合深度的不同:

- 知识驱动型: 检索过程主要或完全依赖于知识图谱. 这类方法直接在图上进行查询与推理, 适用于需要强逻辑约束与可解释性的任务.

- 索引驱动型: 将知识图谱的结构信息融入到文本索引. 例如用邻居实体/关系充当元数据特征, 或将子图摘要拼接到文本后在向量化, 以提升召回与重排效果.

- 混合型: 同时使用图检索与文本检索, 并对结构统一重排与融合:

    - 简单事实 query 可优先图检索; 叙事性/描述性问题可补充文本证据.

    - 复杂 query, 可先在图上定位路径, 再以路径节点为“导航枢纽“做针对性文本检索.

优劣对比:

- 知识驱动: 高精度和可解释性, 但覆盖与容错受限于图谱的完备性.

- 索引驱动: 集成成本低, 但显式推理弱, 可解释性一般.

- 混合型: 综合表现更稳健, 但系统复杂度与工程成本较高.

## 3 前沿 GraphRAG 框架介绍 (截至2025)

### 3.1 GraphRAG (Microsoft)

一个重量级的框架, 核心思想是“先构建全局知识, 再按需检索“. 典型流程:

- 文本 -> 三元组 / 子图构建 -> 图谱 (属性图).

- 采用社区检测进行图划分, 按层级生成全局/社区/局部摘要.

- query 时先匹配全局/社区摘要, 再下钻到局部子图与原文证据.

- 根据需要返回“全局观”与“证据路径“并行的双视角答案.

该方法的优势是提供强全局感知与良好可解释性, 适合探索性分析, 全景总结与多跳问答; 但前期构建与分层摘要成本较高, 对数据持续变更的场景需设计批处理/增量更新与缓存策略.

### 3.2 LightRAG

旨在实现轻量, 高效, 易扩展:

- 核心思路常见为“双层检索”与“图增强索引”, 以较低构建成本获得全局-局部兼顾的检索能力.

- 更强调将结构信号嵌入到文本索引与重排中, 弱化复杂的社区发现流程.

- 适用于资源受限, 快速迭代与在线更新场景.

### 3.3 FRAG (Flexible RAG)

强调灵活性与模块化, 以适应不同复杂度的查询:

- 查询分流: 通过分类器或规则将请求判定为简单/复杂.

- 简单查询: 直接实体链接+属性查找, 低延迟返回.

- 复杂查询: 激活路径检索/多跳推理模块, 再融合文本证据.

- 易扩展: 模块化设计与预训练LLM能力复用, 迁移成本低.

### 3.4 GraphIRAG (Iterative Knowledge Retrieval)

引入“迭代检索“思想, 认为单次检索不足以解决复杂问题:

- 由控制器在生成中多轮触发图查询, 逐步补齐证据链.

- 以“新信息增益”为准则决定是否继续迭代与何时停止.

- 对时间敏感与多跳推理任务更具鲁棒性.


## 4 性能评估与基准测试

### 4.1 核心评估指标

GraphRAG 的评估体系是多维度的, 主要涵盖 3 个方面.

检索质量:

- 传统: Precision、Recall、F1、命中率（Hit Rate@K）。

- RAG特有：
    Context Precision = 检索到的上下文中“真正相关”的比例.

    Context Recall = 所有应当支持答案的“金标准证据”被检回的比例.

    引用/归因准确率 (Citation/Attribution) = 生成答案中的关键断言是否被检索证据正确支撑.

生成质量:

- 问答: 精确匹配 (EM), F1.

- 摘要: ROUGE.

- 事实一致性/忠实度 (Faithfulness): 断言与证据的一致性, 可配合“逐句打标+归因检查”.

系统性能:

推理延迟 (Latency) 是从接收查询到返回最终答案所需的总时间, 是衡量系统响应速度的核心指标. 吞吐量 (Throughput) 指系统在单位时间内能处理的查询数量 (QPS). 成本 (Cost) 包括API调用次数, 计算资源(GPU/CPU)和内存消耗等.

### 4.2 常用基准数据集

学术界已经建立了一系列基准数据集来评测GraphRAG在不同任务上的能力.

- 多跳问答数据集要求模型在多个知识源之间进行推理, 是检验GraphRAG核心能力的试金石. 代表性数据集包括 HotpotQA、2WikiMultihopQA 和 MuSiQue.

- 复杂问答数据集包含结构复杂的查询, 代表性数据集包括 WebQSP 和 ComplexWebQuestions (CWQ).

- 知识图谱驱动的 QA 是专门为评估知识图谱问答能力设计的数据集, 如 KGQAgen-10k.

不过, 一些研究指出, 部分现有基准 (如WebQSP) 存在数据质量问题或评估方法过于僵化 (如严格的EM), 可能导致对模型性能的低估. 因此, 构建更高质量, 更科学的基准是未来的一个重要方向.


## 5 生产环境部署实践与挑战

将GraphRAG框架从实验室推向生产环境, 面临着一系列独特的工程和技术挑战.

- 知识图谱的构建与动态维护: 高质量知识图谱的构建本身就是一项耗时耗力的知识工程. 在生产环境中, 知识需要持续保持最新状态, 如何设计高效, 准确, 低成本的动态更新机制是核心难点.

- 系统性能与可扩展性: 随着知识库规模和用户查询量的增长, 系统的响应延迟, 吞吐量和资源消耗成为主要瓶颈. 生产部署需要应对向量化速度慢, 内存溢出, API 失败等具体问题.

- 安全与隐私保护: RAG 系统引入外部数据源, 带来了新的安全风险, 包括数据隐私泄露, 模型被注入恶意数据 (模型中毒), 以及针对检索模块的攻击等.

- 成本控制: 大规模部署 GraphRAG 系统需要大量的计算和存储资源, 如何优化系统架构以降低运营成本是一个重要的商业考量.
